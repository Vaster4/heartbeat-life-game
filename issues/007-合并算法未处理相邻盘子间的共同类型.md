# Issue #007: 合并算法参与组重分配逻辑问题

## 状态：已修复 ✅

## Commit: ae6c483 (第一次修复), pending (第二次修复 - 吸收模型重写)

## 问题描述

### 原始问题（第一次）

参与组构建逻辑问题，无关盘子被错误拉入参与组。
通过迭代式 BFS 修复，要求邻居必须与直接相邻的组内成员有共同类型。

### 第二次问题：pair-by-pair 重分配策略缺陷

修复 BFS 后，`redistributeGroup` 采用逐对相邻转移策略。
当参与组有 3+ 个盘子通过同一类型连通时，先处理的 pair 会把共同类型转走，
导致后续 pair 找不到共同类型，无法完成全局归集。

```
实际场景：
- (3,1): plate-2 [2,1] ts=2
- (4,1): plate-3 [0,1] ts=3  ← 刚放置
- (4,2): plate-1 [1]   ts=1

三个盘子通过类型 1 连通。
Pair (4,1)↔(4,2) 先处理：plate-3 的 1 → plate-1。plate-3 变成 [0]。
Pair (3,1)↔(4,1) 后处理：plate-3=[0], plate-2=[2,1]，无共同类型，跳过。
结果：plate-2 的 1 没有归集到 plate-1。
```

## 根本原因

pair-by-pair 策略有顺序依赖：先处理的 pair 改变了盘子内容，
导致后续 pair 的共同类型判断失效。

## 最终修复方案

改用全局池化重分配：

1. 收集参与组内所有酒杯到池中，按类型统计数量
2. 盘子按时间戳升序排列（最老优先）
3. 类型按数量降序排列
4. 每种类型独占一个盘子分配（不超过6上限）
5. 类型数 > 盘子数时，最后一个盘子承载剩余类型
