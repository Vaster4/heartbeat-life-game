# Issue #007: 合并算法参与组构建逻辑问题

## 状态：已修复 ✅

## Commit: ae6c483

## 问题描述

### 原始问题（game-log-1773296234991.txt）

当新盘子放入后，如果新盘子的相邻盘子之间有共同类型，但新盘子本身不包含该类型，合并算法不会将这些相邻盘子纳入参与组。

```
棋盘状态：
- (3,0): plate-1 [1, 3] - 包含类型 3（菱形）
- (3,1): 空
- (3,2): plate-3 [3, 2] - 包含类型 3（菱形）

操作：放置 plate-9 [5, 6] 到 (3,1)
```

经过重新分析，这个场景中 (3,1) [5,6] 与 (3,0) [1,3] 和 (3,2) [3,2] 都没有共同类型，
(3,0) 和 (3,2) 虽然都有类型 3 但不直接相邻（中间隔着 (3,1)），
按照"相邻且有共同类型才合并"的规则，不合并是正确行为。

### 第一次修复引入的新问题（game-log-1773297491376.txt）

第一次修复采用"初始组 = 当前盘子 + 所有邻居"的策略，导致无关盘子被错误拉入参与组。

```
棋盘状态（plate-17 放置前）：
- (2,2): [1,1,1]
- (3,2): [7,7,7,7]
- (4,2): 空
- (5,2): [6,6]
- (4,3): [4]

操作：放置 plate-17 [7] 到 (4,2)

错误行为：扫描到 (3,2) 时，初始组 = {(3,2), (2,2), (3,1), (3,3), (4,2)}，
(3,2) 和 (4,2) 共享类型 7 → 整个初始组参与重分配，
导致 (2,2) [1,1,1] 被错误纳入，酒杯被打乱。
```

## 根本原因

第一次修复中，`performOnePass` 无条件将当前盘子的所有邻居纳入初始组，
然后只要组内任意两个盘子有共同类型就触发重分配。
这导致与共同类型完全无关的邻居盘子也被拉入合并。

## 最终修复方案

采用迭代式 BFS 构建参与组：

1. 从当前盘子出发，维护组内所有酒杯类型的集合 `groupTypes`
2. 检查邻居是否与 `groupTypes` 有交集（即与组内任何成员有共同类型）
3. 如果有，纳入组，更新 `groupTypes`，继续检查新成员的邻居
4. 重复直到没有新成员加入

这样既能处理"A 和 C 通过 B 间接连通"的场景（只要每一步都有共同类型），
又不会把完全无关的邻居拉入参与组。

## 修改文件

- `src/core/merge.ts` — `performOnePass` 方法简化，`buildMergeGroup` 方法重写为迭代式 BFS
